ROS►机器人导航

使用AprilTag进行机器人导航（二）

机器人和所有坐标系的情况是这样的:

环境中有4个坐标系：apriltag坐标系、camera坐标系、map坐标系、robot_base坐标系。
apriltag的位置是固定的，三个轴也是不变的；camera装在机器人上，位置和坐标系是变化的；map坐标系是固定的；robot_base就是机器人的中心，这个坐标系也是变化的。

思路1

我们要实现的是从导航点移动到充电点，期间要保证摄像头一直识别到Tag，这实际是一个短距离、小范围的单一路径规划。上一节提到的apriltag2_ros包发布了话题tag_detections，重点是Pose和Orientation

如果没有探测到Tag，话题输出是这样的:

摄像头到Tag的有效距离为0.7m~4.5m，当机器人接触充电桩时，距Tag1m左右（也就是Pose.z为1）。这个思路是这样：

  1. 将机器人导航至距Tag一定距离，摄像头在机器人后部，机器人背对Tag
  2. 订阅话题/tag_detections，检查Tag相对摄像头的x坐标，如果其绝对值较大，转动机器人保证x在0~0.005范围内
  3. 让机器人后退，同时检查x和充电状态，若x不在0~0.005范围内，停止后退，转动机器人直至这个范围内，然后继续后退
  4. 当充电状态为1时，停止机器人

这个程序不复杂，就是订阅话题，在回调函数里判断x。但实际发现了问题：

  1. 两个坐标系的y都是竖直方向，理论上应当是一个不变的差值，但y坐标竟然是变化的
  2. 从不同位置导航，结果发现最终停下的地方差别很大，但x都是0

最终经过分析，问题1的原因就是安装不完善。问题2其实也不难，只是开始没想到。话题tag_detections体现的是Tag坐标系和摄像头坐标系的相对关系，由于apriltag2_ros包让camera坐标系为parent，tag_9坐标系为child，就是在表达Tag在camera坐标系中的坐标，但是有多种情况可以让x坐标为0，也就是这样：

图中的情况都符合x=0，由于摄像头有视角范围，所以符合x=0情况的摄像头轨迹是一段弧线，因此不能用x=0判断机器人导航到位。

思路2

这个思路是把两个坐标系颠倒一下，改为获得摄像头在Tag坐标系中的坐标，因为这样比较符合习惯。当x坐标为0并且机器人的朝向符合条件时，确定机器人达到充点电。
但是经过试验，发现这个思路也不行，看下图：

只用一个Tag时，机器人初始位置应当有所限制。总存在一些位置，当机器人处于这些位置时，无论怎么走，都不能既保持摄像头探测到Tag同时还达到电点。比如下面这种情况，机器人初始位置的X坐标太大，Z坐标太小：

经过试验，初始位置的合理范围比较小。只要在初始位置有一定偏差，导航中途就有探测不到摄像头的可能。但是反过来，如果初始位置比较好，正对充电点，那么干脆不需要Tag导航，让机器人直线运动，根据充电状态停止即可。所以这个思路也不好。

另外经过坐标转换后，数据太不稳定，给导航带来很大困难，只好放弃。

思路3

之前的思路对初始位置都有比较大的限制，于是想到用两个Tag，这样可以扩大有效的初始位置，当然Tag越多效果越好，但策略就越复杂。因为公司需要保密，所以不能详细写出。

问题1：未检测到tag时，程序报错退出


使用try catch，报错就不会退出

问题2：网络状况不好时，程序报错退出

我们调用了函数waitForTransform("camera","tag_9",ros::Time(0),ros::Duration(1));，意思是只等待两个坐标系转换1秒，这在网络情况差时有可能还无法完成转换，rosrun tf tf_echo命令一般都需要几秒时间才能出结果。所以参数最好设置大一点

问题3：检测到Tag9的瞬间，出现段错误后退出

从未检测到两个Tag的状态到检测到Tag9的瞬间，直接段错误而退出。整个过程，我一直看着图像，没发现有什么异常，rostopic echo的结果也是正常的。这个应该就是越界的错误，消息中的detections类型是个数组，在临界状态，其实数组还是空，但调用了msg->detections[0]，所以可以加个判断:

在此之前已经运行了apriltag2_ros的节点，waitForTransform函数中指定了坐标系转换失败前的阻塞时间，这个时间不能太小，否则有可能捕捉不到转换。但设置太大又会影响后面程序中对机器人朝向和运动的判断，因为数据平时浮动就很大。解决方法也简单，不用这么复杂的判断，直接用if(!(msg->detections.empty()))就可以了。

问题5：机器人与充电桩的距离只差一点，还是没接触

后来发现识别二维码获得的坐标与相机的内参有关系，也跟相机标定有关，有时会只差一点，结果导致接触失败。最彻底的方法是把二维码尺寸稍微减小，就算内参有所偏差，不可能差几厘米
